# Phase 7 -- UI Design Contract

> Visual and interaction contract for deep-linkable anchors. Generated by gsd-ui-researcher, verified by gsd-ui-checker.

Phase 7 is ~90% behavioral (History API, scroll mechanics, ResizeObserver pipeline). The visual surface is small: scoping existing anchor CSS from h2-h6 down to h2-h3, and adding `scroll-margin-top` to those headings. No new components, no new colors, no new typography.

## Component Interaction Contracts

### Heading Anchor -- Click (ANCH-01, D-01, D-02)

| State | Visual Behavior |
|-------|----------------|
| Default | No visible `#` symbol (opacity: 0) |
| h2/h3 hover | `#` fades in to the left of heading text at `left: -1.5em`, `--color-accent` (#333333), `opacity: 0 -> 1` over `0.15s ease` |
| h4/h5/h6 hover | **No anchor visible -- h4-h6 anchors removed entirely** (D-05, D-06) |
| Anchor focus (keyboard) | `#` shows at opacity 1 (existing `.heading-anchor:focus` rule, scoped to h2/h3 only) |
| Anchor click | URL updates to `?section={id}` via `pushState`. Viewport smooth-scrolls to the heading. **No clipboard copy. No toast. No visual feedback** beyond URL bar change + scroll. (D-01) |

CSS change: Remove `h4:hover .heading-anchor`, `h5:hover .heading-anchor`, `h6:hover .heading-anchor` from the hover-reveal rule. Final rule:

```css
h2:hover .heading-anchor,
h3:hover .heading-anchor,
.heading-anchor:focus {
  opacity: 1;
}
```

JS change: Anchor injection selector narrows from `h2, h3, h4, h5, h6` to `h2, h3`.

### Deep-Link Load Scroll (ANCH-02, D-07, D-08)

| Scenario | Behavior |
|----------|----------|
| URL has `?section={id}`, heading exists | Page renders at top, then performs a brief smooth scroll to the target heading. The heading lands fully visible below the sticky header with `0.75rem` breathing room. First load uses `replaceState` (D-03). |
| URL has `?section={id}`, heading does NOT exist | Page stays at article top. The `?section=` param is silently stripped from the URL via `history.replaceState`. No error message, no toast, no console warning to the user. (D-08) |
| URL has no `?section=` | Normal article load. No scroll logic triggered. |

Scroll method: `element.scrollIntoView({ behavior: 'smooth' })`. The scroll curve and duration are browser-native (typically ~400-600ms, ease-out). No custom JS animation or CSS `scroll-behavior` override. The `scroll-margin-top` on the heading handles the sticky-header offset natively.

### Browser Back/Forward (ANCH-04, D-02, D-03, D-04)

| Action | Behavior |
|--------|----------|
| User clicks anchor on h2/h3 | `history.pushState` adds `?section={id}` to the history stack |
| User presses Back | `popstate` fires. Handler reads `?section=` from `location.search`. If present, smooth-scrolls to that heading. If absent (user backed out of all section states), no scroll action. Article DOM is never remounted. |
| User presses Forward | Same `popstate` logic -- re-reads `?section=` and scrolls. |

### Sticky-Header Offset Pipeline (ANCH-03, D-09, D-10)

| Component | Responsibility |
|-----------|---------------|
| ResizeObserver (on sticky header element) | Publishes `--header-height` CSS custom property on `:root` (or equivalent scope) whenever the header's `borderBoxSize` changes. Value is the header's current pixel height. |
| h2, h3 elements (in article CSS) | Consume via `scroll-margin-top: calc(var(--header-height, 0px) + 0.75rem)`. The `0px` fallback handles the case before ResizeObserver fires. |
| Browser scroll engine | Uses `scroll-margin-top` natively when `scrollIntoView` is called. No JS offset math needed. |

The `--header-height` variable is the pipeline Phase 8 will consume when the header content changes (home = search bar, article = logo only). The ResizeObserver absorbs those height changes automatically.

## Scroll-Margin-Top CSS Contract

New CSS rule to add to article styles (or equivalent scoped location):

```css
h2, h3 {
  scroll-margin-top: calc(var(--header-height, 0px) + 0.75rem);
}
```

This rule must coexist with the existing `h2, h3` block in `article.css` (which sets font-size, font-weight, line-height, margins, position). It can be added to that same rule block or declared separately.

## Accessibility Contract

| Concern | Decision |
|---------|----------|
| Anchor `aria-hidden` | `"true"` -- the `#` symbol is a supplementary navigation affordance, not the heading text itself. Screen readers skip it and read the heading directly. (Existing behavior, unchanged.) |
| Anchor keyboard access | `.heading-anchor:focus { opacity: 1 }` already present. Tab-navigable. Click handler fires on Enter/Space via native `<a>` element behavior. |
| Focus on deep-link scroll | **Not implemented in Phase 7.** CONTEXT.md deferred section explicitly notes this as a future a11y pass item. The heading is scrolled into view but does not receive programmatic focus. Screen reader users landing on a deep link will be at the page top, not the target heading. |
| Missing-target silence | Acceptable -- no error announcement needed. The URL is cleaned up silently (D-08). The page is at article top, which is the expected default. |
