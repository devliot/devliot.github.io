**Replace the entire method with:**
```typescript
private _scrollToSectionFromUrl(): void {
  const params = new URLSearchParams(window.location.search);
  const section = params.get('section');
  if (!section) return;

  const article = this.shadowRoot?.querySelector('article');
  if (!article) return;

  const target = article.querySelector<HTMLElement>(`#${CSS.escape(section)}`);
  if (target) {
    // D-03: replaceState on initial load so back button leaves cleanly
    history.replaceState({ section }, '', window.location.href);
    target.scrollIntoView({ behavior: 'smooth' });
  } else {
    // D-08: silent miss -- strip invalid ?section= from URL
    const url = new URL(window.location.href);
    url.searchParams.delete('section');
    history.replaceState(null, '', url.toString());
  }
}
```

Key points:
- `CSS.escape(section)` is already in the existing code -- preserve it (T-07-01 mitigation).
- On HIT: `history.replaceState` (NOT pushState) so the initial load does not create a duplicate history entry. The state object `{ section }` is consistent with the pushState calls from the click handler.
- On MISS: `url.searchParams.delete('section')` removes the bad param, `replaceState` updates the URL bar silently. Page stays at article top. No error message, no console warning per D-08.
- `scrollIntoView({ behavior: 'smooth' })` provides the brief smooth scroll per D-07. The `scroll-margin-top` from Plan 01 handles sticky header clearance (ANCH-03).

The `connectedCallback` still calls `_scrollToSectionFromUrl()` -- this handles the case where the component reconnects (though in practice, `updated()` + `updateComplete` is the main entry point after article HTML loads).

private _onPopState = (): void => {
  const section = new URLSearchParams(window.location.search).get('section');
  if (section) {
    const article = this.shadowRoot?.querySelector('article');
    const target = article?.querySelector<HTMLElement>(`#${CSS.escape(section)}`);

## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| URL ?section= param -> DOM query | The ?section= value from the URL is used in querySelector via CSS.escape |
| popstate event -> scroll action | Browser history state change triggers DOM query and scroll |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-07-01 | Spoofing / XSS | `?section=` param used in querySelector | mitigate | `CSS.escape()` wraps the section value before use in `querySelector('#${CSS.escape(section)}')`. This prevents CSS selector injection from crafted URL params. Already present in existing code, preserved in this plan. |
| T-07-02 | Tampering | DOM id derivation | mitigate | Existing `replace(/[^\w-]/g, '')` strips unsafe chars. Unchanged. |
| T-07-03 | Denial of Service | History flood from automated anchor clicks | accept | Requires user gesture (click event). Browser manages history stack depth. Not exploitable without prior script injection. |
</threat_model>
